# 半导体供应链完整本体建模方案 v2.1（修订版）

> 基于29张数据库表的完整业务语义建模
> 覆盖：主数据、供应链、计划、执行、库存、质量、监控全业务域
> 设计理念：本体是企业的数字孪生

---

## 一、29张表的完整业务语义分析


### 1.1 业务域划分

通过分析你的仿真逻辑，29张表可以分为**7大业务域**：

```
┌─────────────────────────────────────────────────────────┐
│                  半导体制造业务全景                        │
├──────────┬──────────┬──────────┬──────────┬──────────────┤
│ 主数据域 │ 供应链域 │  计划域  │  执行域  │   库存域     │
│ (6张表)  │ (4张表)  │ (5张表)  │ (5张表)  │  (3张表)     │
├──────────┴──────────┴──────────┴──────────┴──────────────┤
│                    质量域 (2张表)                          │
├──────────────────────────────────────────────────────────┤
│                  监控域 (4张表)                            │
└──────────────────────────────────────────────────────────┘
```

### 1.2 每张表的实际含义

#### 域1：主数据域 (Master Data) - 6张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **product** | 产品定义 | 工厂生产什么（100个BGA/QFN/SOP产品） | ✅ 核心 |
| **material** | 物料定义 | 生产需要什么（Die、基板、EMC等） | ✅ 核心 |
| **work_center** | 工作中心 | 产能资源池（25个工作中心） | ✅ 核心 |
| **machine** | 机台设备 | 具体生产设备（96台机台） | ✅ 核心 |
| **shift_pattern** | 班次模式 | 工作时间定义（日班/夜班） | ⚠️ 辅助 |
| **process_route** | 工艺路线 | 产品怎么生产（80-120道工序） | ✅ 核心 |
| **route_step** | 工序定义 | 工艺路线的具体步骤 | ✅ 核心 |
| **machine_capability** | 机台能力矩阵 | 哪台机台能做什么产品 | ✅ 核心 |
| **setup_matrix** | 换线矩阵 | 产品切换需要多长时间 | ✅ 核心 |

**业务逻辑**：
```
Product (产品) 
  → has_route → ProcessRoute (工艺路线)
    → has_steps → RouteStep (工序)
      → requires → WorkCenter (工作中心)
        → has_machines → Machine (机台)
          → capable_of → MachineCapability (能力矩阵)
            → setup_between → SetupMatrix (换线时间)
```

---

#### 域2：供应链域 (Supply Chain) - 4张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **supplier** | 供应商 | 谁能供货（13个主备供应商） | ✅ 核心 |
| **supplier_material** | 供应商-物料关系 | 谁能供什么（价格、交期、MOQ） | ✅ 核心 |
| **material_substitute** | 物料替代关系 | 缺料时可用什么替代 | ✅ 核心 |
| **bom** | 物料清单 | 产品由哪些物料组成 | ✅ 核心 |

**业务逻辑**：
```
Supplier (供应商)
  → supplies → Material (物料)
    → substitutes → Material (替代料)
  
Product (产品)
  → has_bom → Material (物料)
    → supplied_by → Supplier (供应商)
```

---

#### 域3：计划域 (Planning) - 5张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **customer_order** | 客户订单 | 市场需求输入 | ✅ 核心 |
| **work_order** | 工单 | 生产计划（由订单生成） | ✅ 核心 |
| **work_order_operation** | 工单工序 | 工单的工序级计划 | ✅ 核心 |
| **work_order_material** | 工单物料需求 | 每个工单需要哪些物料 | ✅ 核心 |
| **purchase_order** | 采购订单 | 向供应商下单 | ✅ 核心 |
| **purchase_order_line** | 采购订单行 | 采购的物料明细 | ⚠️ 从属 |

**业务逻辑**：
```
CustomerOrder (客户订单)
  → generates → WorkOrder (工单)
    → has_operations → WorkOrderOperation (工单工序)
      → requires → WorkOrderMaterial (物料需求)
        → triggers → PurchaseOrder (采购订单)
          → has_lines → PurchaseOrderLine (采购行)
```

---

#### 域4：执行域 (Execution) - 5张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **wip_lot** | 在制品批次 | Lot级别的生产追踪（25片/批） | ✅ 核心 |
| **production_task** | 生产任务 | 机台级别的任务执行记录 | ✅ 核心 |
| **material_transfer** | 物料调拨 | 工单间的物料挪用 | ⚠️ 辅助 |
| **work_calendar** | 工作日历 | 每天/每工作中心是否工作 | ⚠️ 辅助 |

**业务逻辑**：
```
WorkOrderOperation (工单工序)
  → creates → WIPLot (在制品批次)
    → assigned_to → ProductionTask (生产任务)
      → runs_on → Machine (机台)
  
WorkOrderMaterial (工单物料需求)
  → may_trigger → MaterialTransfer (物料调拨)
  
WorkCalendar (工作日历) 约束 → ProductionTask (生产任务)
```

---

#### 域5：库存域 (Inventory) - 3张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **inventory** | 原材料库存 | 物料的可用量、预留量 | ✅ 核心 |
| **inventory_transaction** | 库存事务流水 | 所有库存变动的审计日志 | ⚠️ 流水 |
| **finished_goods_inventory** | 成品库存 | 完工产品的库存 | ✅ 核心 |

**业务逻辑**：
```
Material (物料)
  → has_inventory → Inventory (原材料库存)
    → tracked_by → InventoryTransaction (事务流水)

Product (产品)
  → has_fg_inventory → FinishedGoodsInventory (成品库存)
```

---

#### 域6：质量域 (Quality) - 2张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **quality_inspection** | 质量检验记录 | IQC/IPQC/FQC检验结果 | ✅ 核心 |

**业务逻辑**：
```
ProductionTask (生产任务)
  → triggers → QualityInspection (质量检验)
    → result → Pass/Rework/Scrap

PurchaseOrder (采购订单)
  → triggers → QualityInspection (IQC检验)
```

---

#### 域7：监控域 (Monitoring) - 4张表

| 表名 | 实际含义 | 业务角色 | 是否核心对象 |
|-----|---------|---------|------------|
| **schedule** | 排程汇总 | 每天的产能负荷统计 | ⚠️ 聚合 |
| **machine_status_log** | 机台状态日志 | 机台运行/停机/维护记录 | ⚠️ 流水 |

**业务逻辑**：
```
ProductionTask (生产任务)
  → generates → MachineStatusLog (机台日志)
  → aggregates → Schedule (排程汇总)
```

| **总计** | **65个** | **完整覆盖全业务语义** |

### 4.2 关系设计原则

**本体关系 ≠ 数据库外键**

本体关系基于业务语义定义，不仅限于数据库外键。例如：

- **中间表的多重关系**：BOM表有3个外键（product_id, material_id, step_id），在本体中表示为4个关系：
  - R3: Product ↔ Material (N:N，通过BOM)
  - R3_1: Product → BOM (1:N)
  - R3_2: BOM → Material (N:1)
  - R3_3: RouteStep → BOM (1:N)

- **业务语义关系**：即使没有外键，只要有业务关联就定义关系
  - 如：Machine → ProductionTask (1:N)，虽然ProductionTask.machine_id是外键，但这是执行层面的重要语义
